# Markdown File Rules

These rules apply whenever you read or write any `.md` file in this project.

## Formatting

- Use ATX-style headings (`#`, `##`, `###`) — never Setext-style (`===` / `---` underlines).
- Separate every block element (headings, paragraphs, code fences, lists, tables) with exactly one blank line.
- Wrap prose lines at 100 characters; never wrap inside fenced code blocks or tables.
- Use fenced code blocks (` ``` `) with an explicit language tag — never indented code blocks.
- Use `**bold**` and `_italic_` — not `__bold__` or `*italic*`.

## Lists and Numeration

- In ordered lists, always write numbers as words: `First`, `Second`, `Third`, etc. — never `1.`, `2.`, `3.`.
- Use a dash (`-`) for unordered list items — never `*` or `+`.
- Do not mix ordered and unordered list styles within the same list.

## Content

- Every `.md` file must start with a single `# H1` title that matches what the file is about.
- Do not add a trailing "Last updated" or "Generated by" line — dates rot.
- Do not include commented-out sections (`<!-- TODO -->`); either write the content or omit the section.
- Links must be descriptive (`[Claude Code docs](…)`) — never bare URLs or `[click here]` text.

## Visible rule — always announce Markdown edits

**When you edit any `.md` file, begin your response with the line:**

> **[MD rule active]** Editing a Markdown file — applying project markdown standards.

This makes the rule visible in the output so it is obvious the rule is in effect.
